Diário de Sprint - Classroom Connect

Divisão das responsabilidades

Durante o desenvolvimento do projeto, realizamos a divisão das atividades considerando as principais entregas da atividade.

As responsabilidades foram organizadas da seguinte forma:

- Levantamento e organização dos requisitos:
  Responsável por analisar o problema, definir o público-alvo, objetivos da solução e elaborar as User Stories.

- Definição do MVP e funcionalidades:
  Responsável por identificar as funcionalidades essenciais do sistema e separar funcionalidades futuras.

- Modelagem UML:
  Responsável pela criação e organização do diagrama de casos de uso conforme as User Stories e funcionalidades do MVP.

- Documentação do projeto:
  Responsável pela organização das informações no repositório, criação dos documentos e padronização das entregas.

- Gerenciamento do projeto:
  Responsável pelo acompanhamento das tarefas utilizando o GitHub Projects, organização do Kanban e controle do andamento das atividades.



Principais decisões tomadas

Durante a sprint, algumas decisões foram definidas para manter o projeto coerente com o objetivo do sistema:

- Definição do Classroom Connect como uma plataforma de gerenciamento acadêmico para centralizar atividades, turmas, avaliações e comunicação.

- Priorização das funcionalidades essenciais para composição do MVP:
  login, gerenciamento de turmas, publicação de atividades, envio de trabalhos, lançamento e consulta de notas.

- Separação das funcionalidades futuras para evitar aumentar a complexidade inicial do sistema, mantendo o foco no funcionamento principal.

- Definição dos atores do sistema como estudante, professor, administrador e coordenador, considerando suas responsabilidades dentro da plataforma.

- Criação do diagrama de casos de uso baseado nas User Stories e funcionalidades do MVP.

- Escolha do GitHub Projects como ferramenta de gerenciamento por permitir organização do backlog, quadro Kanban e acompanhamento das tarefas.



Dificuldades encontradas

Durante o desenvolvimento do projeto, algumas dificuldades foram identificadas:

- Definir quais funcionalidades realmente deveriam fazer parte do MVP e quais poderiam ser consideradas futuras.

- Relacionar corretamente as User Stories com os casos de uso do diagrama UML.

- Definir os atores e suas permissões para evitar conflitos entre as responsabilidades de cada usuário.

- Aplicar corretamente os conceitos de UML, principalmente o uso do relacionamento <<extends>>.


Soluções adotadas

Para resolver as dificuldades encontradas, foram realizadas algumas ações:

- Revisão das User Stories e funcionalidades para garantir que o MVP representasse apenas os recursos essenciais do sistema.

- Análise dos papéis de cada usuário para definir corretamente suas ações dentro do diagrama de casos de uso.

- Organização do diagrama considerando somente funcionalidades essenciais, deixando funcionalidades futuras fora do MVP.

- Utilização do GitHub Projects para acompanhar o progresso das tarefas e melhorar a organização da equipe.

- Revisão das decisões tomadas para garantir que todas as partes do projeto permanecessem alinhadas com o objetivo inicial.
